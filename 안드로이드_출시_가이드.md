# 안드로이드 출시 가이드

이 문서는 `lotto_recommend_app`을 **독립 안드로이드 앱(구글플레이)**으로 출시하기 위해
무엇을 고쳤고, 무엇을 더 해야 하는지 정리한 것입니다. 코드를 수정한 사람(Claude)이
직접 실행/배포까지 해줄 수는 없으므로(서버 호스팅, 계정 가입, 키 발급 등은 본인 명의로
진행해야 함), 아래 순서를 따라가면 됩니다.

---

## 0. 가장 먼저: 보안 — 노출된 키 교체

`.project-config.json` 파일에 **실제 DB 비밀번호, JWT 서명 키, AWS 임시자격증명**이 평문으로
들어있었습니다. 이미 마누스에게 공유되었던 값이라 더 이상 신뢰할 수 없습니다.

- [ ] DB 비밀번호를 새로 발급(rotate)하세요. (기존 TiDB Cloud 인스턴스를 계속 쓸 경우)
- [ ] `JWT_SECRET`을 새 무작위 문자열로 교체하세요 (`openssl rand -base64 48`).
- [ ] 이 두 값을 포함한 모든 비밀값은 `.env` 파일에만 두고, **절대 git에 커밋하지 마세요.**
- [ ] `.project-config.json`은 이미 평문 비밀값을 제거해 두었습니다. 그래도 git 히스토리에
      과거 커밋이 남아있다면, 저장소를 새로 만들거나 히스토리를 정리하는 것을 권장합니다.

## 1. 이번에 코드에서 고친 것 (요약)

| 영역 | 이전 상태 | 수정 후 |
|---|---|---|
| 로그인 | "토스 계정으로 시작하기"라고 써있지만 실제로는 Manus 자체 인증 서버(`manus.im`)에 연결됨 — 배포 시 작동 안 함 | 실제로 동작하는 **카카오 로그인**으로 교체 |

| 무료체험 | "가입 후 1주, 1회 10개 조합 무료"가 코드에 없었음 | `subscription.claimTrial` 추가 (구독 이력이 없는 사용자만 1회) |
| 번호 배분 동시성 | 여러 사용자가 동시에 요청하면 같은 번호가 두 명에게 나갈 수 있는 레이스 컨디션 | DB 트랜잭션 + 행 잠금(`SELECT...FOR UPDATE`)으로 직렬화 |
| 부당광고 방지 | 결제 화면에 면책 문구가 약하게 노출 | 결제 화면에 "당첨 확률을 수학적으로 높이지 않는다"는 문구를 눈에 잘 보이게 항상 노출 |
| 아이콘/애널리틱스 | Manus 자체 스토리지·애널리틱스에 의존 (독립 배포 시 깨짐) | 로컬 아이콘 경로로 변경 (아이콘 파일은 직접 채워야 함), Manus 애널리틱스 스크립트 제거 |

테스트도 일부 손봤습니다 (`server/allocation.flow.test.ts`를 DB 의존 없는 순수 로직 테스트로
재작성, `server/plans.logic.test.ts`의 무료체험 플랜 관련 깨진 단언 수정). **이 환경에는 Node.js가
없어 직접 `pnpm test`를 실행해보지 못했습니다.** 본인 환경에서 한 번 실행해서 확인해주세요.

## 2. 직접 진행해야 하는 설정들

### 2-1. 카카오 로그인 등록
1. https://developers.kakao.com 에서 애플리케이션 생성
2. "카카오 로그인" 활성화
3. Redirect URI 등록: `https://yourdomain.com/api/oauth/callback`
4. 동의 항목에서 닉네임, 이메일(선택) 설정
5. 발급받은 **REST API 키**를 `.env`의 `KAKAO_CLIENT_ID`, `VITE_KAKAO_CLIENT_ID`에 동일하게 입력

### 2-2. 서버 호스팅
이 앱은 React(Vite) + Express(tRPC) + MySQL 풀스택 앱입니다. 직접 운영 가능한 서버가
필요합니다 (예: 가벼운 VPS, Railway/Fly.io 같은 PaaS, 또는 국내 클라우드). MySQL은
기존 TiDB Cloud를 계속 쓰거나(비밀번호는 새로 발급), PlanetScale/RDS 등으로 옮겨도 됩니다.

- [ ] `pnpm install` → `pnpm run build` → `pnpm run start` 로 정상 기동되는지 먼저 로컬/서버에서 확인
- [ ] `pnpm drizzle-kit generate && pnpm drizzle-kit migrate` 로 스키마 마이그레이션 적용
      (`subscriptions.purchaseToken` 컬럼 등 새로 추가된 컬럼 포함)
- [ ] HTTPS 필수 (카카오 로그인 redirect, Play Billing 모두 HTTPS 전제)

### 2-3. 아이콘 이미지
기존 아이콘은 Manus 자체 스토리지에 있어 더 이상 접근할 수 없습니다.
`client/public/icons/icon-192.png`, `icon-512.png` 를 직접 준비해서 넣어주세요
(정사각형, 배경 투명/단색 권장 — 안드로이드 적응형 아이콘 가이드 참고).

### 2-4. 구글 플레이 콘솔 + Play Billing
1. https://play.google.com/console 에서 개발자 계정 생성 (최초 1회 $25)
2. 앱 만들기 → 패키지명 결정 (예: `com.yourcompany.lottoapp`) → `.env`의
   `GOOGLE_PLAY_PACKAGE_NAME`에 입력
3. **수익 창출 > 상품 > 구독**에서 상품 2개 등록 (코드의 `playProductId`와 정확히 일치해야 함):
   - `month_sub` — 5,500원 / 1개월 정기결제
   - `year_sub` — 11,000원 / 1년 정기결제
4. **설정 > API 액세스**에서 서비스 계정 생성 → JSON 키 다운로드 →
   해당 서비스 계정에 "재무 데이터 보기/관리" 권한 부여
5. JSON 키 파일 내용 전체를 `.env`의 `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON`에 한 줄로 붙여넣기

### 2-5. 안드로이드 패키징 (TWA / Bubblewrap 권장)
이 앱은 PWA 형태이므로, **Bubblewrap**으로 Trusted Web Activity(TWA) 패키지를 만드는 것이
가장 빠른 길입니다 (구글이 공식 권장하는 PWA → 안드로이드 변환 방법).

```bash
npm install -g @bubblewrap/cli
bubblewrap init --manifest=https://yourdomain.com/manifest.json
# 안내에 따라 패키지명, 키스토어 생성 등 진행
```

생성된 `twa-manifest.json`에서 Play Billing을 쓰려면 아래 설정이 필요합니다:

```json
{
  "features": {
    "playBilling": { "enabled": true }
  },
  "alphaDependencies": { "enabled": true }
}
```

이후:
```bash
bubblewrap update
bubblewrap build
```
→ `app-release-signed.aab` 파일이 생성되며, 이걸 Play Console에 업로드합니다.

또한 **Digital Asset Links** 설정이 필요합니다 (도메인 소유 확인 — TWA 주소창이 안 보이게
하려면 필수). `https://yourdomain.com/.well-known/assetlinks.json`에 Bubblewrap이 알려주는
내용을 올려야 합니다.

> ⚠️ 참고: Play Billing + Digital Goods API 연동은 구글 문서에도 "playBilling 기능 플래그",
> "alphaDependencies" 같은 별도 설정이 필요하다고 명시되어 있고, 실제로 설정이 조금만
> 틀려도 "결제 방법이 지원되지 않습니다" 에러가 나는 경우가 보고되어 있습니다. 결제까지
> 한 번에 다 동작하지 않더라도 당황하지 말고, `assetlinks.json`과 `playBilling` 플래그부터
> 다시 확인하세요.

대안: TWA에서 결제 연동이 계속 막히면, **Capacitor**로 웹뷰를 감싸고 네이티브 Play Billing
플러그인(예: `cordova-plugin-purchase` 계열)을 쓰는 방법도 있습니다. 다만 유지보수할
네이티브 프로젝트가 하나 더 생기는 트레이드오프가 있습니다.

## 3. 부당광고/사행성 관련 체크리스트 (출시 전 필수)

요청하신 "확률이 높은 로또 번호"라는 표현은 **쓰지 않는 것을 강하게 권장**합니다. 로또는
매 회차 독립적인 무작위 추첨이라 과거 데이터 분석으로 당첨 확률을 수학적으로 높일 수
없습니다. 이미 앱 코드 안의 `DISCLAIMER` 상수가 정확히 이 점을 명시하고 있고, 이번에
결제 화면에서도 항상 보이도록 했습니다.

- [ ] 앱 설명, 스크린샷, 마케팅 문구에서 "확률이 높다", "당첨 확률 상승", "1등 보장" 류의
      표현을 모두 제거 — "통계 기반 번호 구성/분석 참고 서비스" 정도로 표현
- [ ] 결제 화면·구독 화면에 면책 문구가 항상 보이는지 확인 (이미 반영함)
- [ ] 개인정보처리방침/이용약관 페이지를 Play Console 등록 시 URL로 제출 (앱 내 `/privacy`, `/terms`)
- [ ] Play Console "콘텐츠 등급 설문"에서 도박/사행성 관련 문항에 정직하게 답변
      (실제 베팅·복권 판매가 아니므로 일반적으로는 통과되지만, 검토가 추가로 걸릴 수 있음을
      감안해 출시 일정에 여유를 두세요)
- [ ] 앱 이름이 "조작 로또번호 분석"으로 되어 있던 부분(헤더, 약관)을 그대로 둘지 다시
      검토해보세요. "조작"이라는 단어가 (실제로는 "운영자가 직접 골랐다"는 뜻이지만)
      복권 조작/부정행위로 오해될 여지가 있어, 심사관이나 사용자에게 불필요한 의심을 살
      수 있습니다. `manifest.json`/페이지 타이틀은 이번에 "로또 통계 분석"으로 바꿔두었고,
      앱 헤더(`AppShell.tsx`)와 약관 문구는 브랜드 결정이라 그대로 남겨두었습니다.

## 5. 아직 남아있는 기술 부채 (시간상 이번에 다루지 못한 것)

- `server/_core/storageProxy.ts`, `server/_core/notification.ts` 등 일부는 여전히 Manus의
  Forge API(`BUILT_IN_FORGE_API_URL/KEY`)에 의존합니다. 사용자 프로필 이미지 업로드나
  운영자 알림 기능을 쓴다면 이 부분도 별도 스토리지(S3 등)로 교체가 필요합니다.
  지금 당장 핵심 기능(로그인/결제/번호배분)에는 영향이 없어 우선순위를 낮췄습니다.
- 광고(AdMob 등)는 여전히 자리만 있는 플레이스홀더입니다.
- 푸시 알림(FCM)도 토큰 저장 자리만 있고 실제 발송 로직은 없습니다.
- 데이터베이스 마이그레이션 파일은 `drizzle-kit generate`로 직접 생성해야 합니다
  (이 환경엔 node가 없어 대신 생성해드리지 못했습니다).

## 6. (추가) 주간 당첨 발표 + 푸시 알림 시스템

오늘 다음 기능을 새로 만들었습니다:

- 매주 일요일 오전 9시(KST), 서버가 자동으로 그 주 배분 조합을 당첨번호와 대조해서
  1~5등 인원수를 집계합니다 (별도 cron 서비스 없이, 서버 안에서 매분 시간을 체크하는
  방식으로 구현 — 추가 비용 없음).
- 전체 회원에게 "1등 N명, 2등 N명..." 요약 푸시를 보내고, 실제 당첨자에게는
  개별 축하 메시지를 추가로 보냅니다.
- 홈 화면에 같은 내용의 공지 배너가 뜨고, "1주일간 보지 않기"를 누르면 서버에 기록되어
  그 주 동안 다시 안 뜹니다.
- 운영자 화면(`/admin` → "당첨 결과" 탭)에 "지금 발표 + 발송하기" 버튼이 있어서,
  자동 실행을 기다리지 않고 수동으로도 테스트/재실행할 수 있습니다. 같은 주차는
  한 번만 발송되도록 막아놔서 중복 클릭해도 안전합니다.

### ⚠️ 아직 안 된 것: 실제 휴대폰이 알림을 받으려면 한 단계가 더 필요해요

서버 쪽(알림을 "보내는" 기능)은 완성했지만, 휴대폰 앱이 "이 알림을 받을게요"라고
등록하는 부분(Firebase JS SDK)은 이번에는 일부러 안 건드렸습니다. 이유는, 그 작업을
하려면 새로운 코드 부품(npm 패키지)을 설치해야 하는데, 지금 이 프로젝트는 부품 목록을
딱 맞게 고정해두는 방식(pnpm-lock.yaml)을 쓰고 있어서, 새 부품을 잘못 추가하면
오늘 어렵게 고친 서버가 또 통째로 안 켜질 위험이 있었습니다. 오늘 너무 많은 걸
한 번에 바꾸면 어디서 문제가 생겼는지 찾기도 어려워지니, 이 부분은 다음 세션에서
차분히 진행하는 것을 권장합니다.

지금 상태로 "지금 발표 + 발송하기"를 눌러보면, 집계(1~5등 인원수)는 정확히 계산되고
화면에도 잘 나오지만, 등록된 휴대폰이 아직 없어서 "전체 발송 0건"으로 나올 거예요 —
정상입니다, 오류 아니에요.
